\section{Постановка задачи}

Необходимо реализовать два алгоритма сортировки массива целых чисел по возрастанию и провести их экспериментальное сравнение. Для каждого алгоритма требуется подсчитать количество сравнений элементов и количество перемещений (обменов).

Сравнение выполняется на одинаковых исходных массивах четырёх типов:
1) элементы уже упорядочены по возрастанию;  
2) элементы упорядочены по убыванию;  
3) расстановка элементов случайна (широкий диапазон значений);  
4) расстановка элементов случайна (узкий диапазон значений, много повторов).

Эксперименты выполняются для размеров $n = 10, 100, 1000, 10000$. В работе сравниваются шейкерная сортировка (\code{shakerSort}) и сортировка выбором (\code{selectionSort}). Корректность результата проверяется сравнением с эталонной сортировкой \code{qsort}.

По результатам экспериментов видно, что шейкерная сортировка существенно зависит от упорядоченности входа: на уже отсортированном массиве обменов нет, а выполнение завершается раньше за счёт флага \code{changed}. На обратном массиве (тип 2) наблюдается близкий к худшему случаю характерный для квадратичной трудоёмкости порядка $O(n^2)$.

Сортировка выбором даёт одинаковое число сравнений на всех типах массивов, поскольку поиск минимального элемента выполняется полностью на каждом шаге независимо от данных. Поэтому число сравнений равно $n(n-1)/2$. При этом перемещений заметно меньше: выполняется не более одного обмена на позицию, и если минимальный элемент уже находится на месте, обмен не производится.

Описания алгоритмов и общие оценки трудоёмкости использовались по источникам~[1, 2].

% ===================== 3. СТРУКТУРА ПРОГРАММЫ =====================
\section{Структура программы и спецификация функций}

Программа написана на языке C и состоит из набора функций генерации входных данных, двух алгоритмов сортировки и кода экспериментального запуска в \code{main}. Подсчёт операций ведётся через указатели на счётчики сравнений и обменов, чтобы функции сортировки не зависели от глобальных переменных.

Ниже приведён полный список реализованных функций и их назначение.

\textbf{\code{unsigned long long rnd(void)}}  
Функция расширяет стандартный генератор \code{rand()}, объединяя несколько его вызовов в одно значение. Это позволяет получать случайные числа с более широким диапазоном, чем один вызов \code{rand()}. В работе эта функция используется при генерации массивов типов 1--4.

\textbf{\code{void gen\_array\_int(int *a, int n, int param)}}  
Генератор массива \code{a} длины \code{n} по параметру \code{param}. Реализованы четыре типа входов:
\code{param = 1} --- массив по возрастанию (почти “лучший” случай для многих методов);  
\code{param = 2} --- массив по убыванию (часто “тяжёлый” случай);  
\code{param = 3} --- случайные значения из большого диапазона;  
\code{param = 4} --- случайные значения из малого диапазона (много одинаковых элементов).

\textbf{\code{int compare(const void *a, const void *b)}}  
Компаратор для \code{qsort}. Нужен для построения эталонного результата сортировки: исходный массив копируется в \code{check}, сортируется \code{qsort}, затем сравнивается с результатами собственных сортировок.

\textbf{\code{void shakerSort(int *a, int n, long long *comp, long long *swaps)}}  
Реализация шейкерной сортировки (двунаправленного пузырька). Алгоритм выполняет проход слева направо, затем справа налево, сужая рабочие границы \code{left} и \code{right}.  
Счётчик \code{*comp} увеличивается перед каждой проверкой порядка соседних элементов. Счётчик \code{*swaps} увеличивается только при реальном обмене двух соседних элементов. Флаг \code{changed} позволяет завершить сортировку досрочно, если за полный цикл не было ни одного обмена.

\textbf{\code{void selectionSort(int *a, int n, long long *comp, long long *swaps)}}  
Реализация сортировки выбором. Для каждой позиции \code{i} ищется индекс минимального элемента на отрезке \code{i..n-1}; затем при необходимости выполняется один обмен элементов \code{a[i]} и \code{a[minI]}.  
Сравнения считаются внутри поиска минимума, обмен считается только если \code{minI != i}.

\textbf{\code{int main(void)}}  
Организует эксперимент: перебирает размеры \code{sizes = \{10, 100, 1000, 10000\}}, выделяет память под массивы \code{orig}, \code{work}, \code{check}. Для каждого типа входа генерирует \code{orig}, строит эталон \code{check} через \code{qsort}, затем запускает \code{shakerSort} и \code{selectionSort} на копиях массива (\code{memcpy}). Корректность проверяется через \code{memcmp}, результаты и статистика печатаются в консоль.

% ===================== 4. ОТЛАДКА/ТЕСТЫ =====================
\section{Отладка программы, тестирование функций}

Отладка выполнялась поэтапно: сначала проверялась генерация входных массивов, затем корректность каждого алгоритма сортировки на небольших данных, после чего добавлялась автоматическая проверка результатов через эталон \code{qsort}. Далее проверялась корректность подсчёта операций (сравнений и обменов), чтобы статистика соответствовала реальным действиям алгоритмов.

Тестирование включает минимум три теста, которые повторялись для обеих сортировок.

\textbf{Тест 1: уже отсортированный массив (тип 1).}  
Цель: убедиться, что алгоритмы сохраняют порядок и корректно считают операции. Для \code{shakerSort} важен ранний выход: если массив отсортирован, после прохода слева направо флаг \code{changed} остаётся равным нулю, и основной цикл завершается. Ожидаемый результат: \code{swaps = 0}.

\textbf{Тест 2: массив в обратном порядке (тип 2).}  
Цель: проверить “тяжёлый” случай и отсутствие ошибок границ циклов. На этом тесте выявляются типичные проблемы: неверные границы \code{left/right}, ошибки в сравнении \code{a[i] > a[i+1]} или \code{a[i] < a[i-1]}, а также некорректный обмен. Ожидается значительное число операций у \code{shakerSort} и существенно меньшее число обменов у \code{selectionSort} (не более одного обмена на позицию).

\textbf{Тест 3: случайные значения (тип 3).}  
Цель: проверить “обычный” случай на широком диапазоне значений, включая отрицательные числа. На этом тесте хорошо видно, что алгоритмы не зависят от конкретных значений, а только от порядка, и корректно обрабатывают любые целые числа.

\textbf{Тест 4: случайные значения с повторами (тип 4).}  
Цель: проверить поведение при большом количестве равных элементов. Это полезно для выявления ошибок в условиях сравнения (например, если по ошибке написать \code{>=} вместо \code{>} и считать лишние обмены). В текущей реализации сравнения сделаны так, что равные элементы не требуют обменов.

\textbf{Автоматическая проверка результата.}  
Для каждого тестового массива создавался эталон: копия \code{orig} сортировалась \code{qsort(check, ...)}. После выполнения \code{shakerSort} и \code{selectionSort} массив \code{work} сравнивался с эталоном \code{check} с помощью \code{memcmp}. В выводе фиксировалось \code{correct = YES/NO}, что позволяет быстро найти ошибку, если сортировка работает неверно.

\textbf{Контроль корректности сравнения алгоритмов.}  
Каждая сортировка запускается на одной и той же исходной выборке: перед вызовом сортировки выполняется \code{memcpy(work, orig, ...)}. Это гарантирует, что второй алгоритм не получает на вход уже отсортированный результат первого.

% ===================== 5. АНАЛИЗ ОШИБОК =====================
\section{Анализ допущенных ошибок}

В процессе написания и проверки программы можно выделить несколько типичных ошибок, которые заметно влияют и на корректность результата, и на “честность” экспериментальных чисел.

\textbf{1) Неинициализированные элементы при генерации.}  
Если в одной из веток генератора вычислить значение, но не присвоить его в \code{a[i]}, часть массива остаётся с неопределёнными данными. Тогда сортировка и подсчёт операций становятся непредсказуемыми. Решение: контролировать, что в каждой ветке \code{gen\_array\_int} выполняется присваивание всем \code{a[i]}.

\textbf{2) Несопоставимые входы для разных сортировок.}  
Если запускать алгоритмы последовательно на одном и том же массиве без копирования, второй алгоритм фактически сортирует уже упорядоченный результат первого. Тогда статистика оказывается неверной. Решение: перед каждым запуском сортировки делать копию исходного массива (в программе используется \code{memcpy}).

\textbf{3) Некорректный подсчёт сравнений и обменов.}  
Счётчики легко “сбить”, если увеличивать их не там, где реально происходит операция: например, считать обмен, даже если перестановки не было, или забыть увеличить число сравнений перед проверкой порядка элементов. Решение: увеличивать \code{*comp} непосредственно перед каждым сравнением элементов, а \code{*swaps} --- только в момент фактического обмена.

После устранения подобных ошибок обе реализации стабильно дают правильный результат на всех типах массивов и всех значениях $n$, что подтверждается сравнением с эталоном \code{qsort}.
